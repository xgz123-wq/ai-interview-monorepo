# AI-Interview-Agent — 从旧版 ai-interview 升级部署指南(保留数据)

> **适用版本**:VM 已部署过旧版 `ai-interview`(9 张表 + 普通 `postgres:16-alpine` 镜像),现在要升级到新版 `ai-interview-agent`(13 张表 + `pgvector/pgvector:pg16` 镜像 + Agent),且**希望保留旧数据**(用户、面试记录、简历等)。
>
> 如果不需要保留旧数据,或旧 VM 上只是测试数据,**直接走 [部署文档.md](./部署文档.md) §第二步**(清场重建,更简单)。
>
> 本文档是 [部署文档.md](./部署文档.md) 的**补充章节**,二者配合使用。

---

## 写在前面

1. 本文档假设 VM 已经按 [部署文档.md §1.1–1.9](./部署文档.md) 完成了基础环境准备(SSH / VMware Tools / 共享文件夹 / Docker / Docker Compose / 镜像加速 / 静态 IP)。如果还没做,先去看主文档。
2. 升级前**必须做 pg_dump 备份**。本文档会反复强调这一点。
3. 新版相对旧版新增了 4 张表 + pgvector 扩展 + Agent 工具调用,数据库 schema 升级涉及迁移链修复、镜像替换两个核心动作,任何一个出错都可能导致数据不可用。**严格按顺序执行**。
4. 升级过程大约需要 **30 分钟**(不含 RAG 数据初始化)。

---

## §1 适用场景与前置条件

### 适用

- ✅ VM 上已部署旧版 `ai-interview`(`docker ps` 能看到 `ai-interview-app` / `ai-interview-postgres` / `ai-interview-redis` 三个核心容器)
- ✅ 旧项目数据库里有真实数据(用户、面试记录、简历),不想丢
- ✅ 旧 PG 镜像是 `postgres:16-alpine` 或其他 PG 16 主版本(**必须是 16,pgvector 镜像只兼容 16**)

### 不适用(请直接走主文档 §第二步清场重建)

- ❌ 旧数据没有价值,只是测试数据
- ❌ VM 上有多个 PostgreSQL 项目混跑,无法停服
- ❌ 旧 PG 镜像是 14/15 主版本(必须升级到 16)
- ❌ 想从其他类似项目迁移过来(本指南只针对 `ai-interview` → `ai-interview-agent` 这一条路径)

## §2 升级前必读:3 条红线

| # | 红线 | 不满足的后果 | 解决 |
|---|------|--------------|------|
| 1 | **postgres 镜像必须替换为 `pgvector/pgvector:pg16`** | 普通 `postgres:16-alpine` 镜像里没有 vector 扩展的二进制文件,`CREATE EXTENSION vector` 会失败,所有 RAG 相关表建不出来 | 新版 `docker-compose.yml` 已用 pgvector 镜像,确认即可 |
| 2 | **必须把旧版的 `3b8a0d9f3f5a` 迁移文件复制到新版 `migrations/versions/`** | 新版迁移链缺少旧版的最后一个迁移,`alembic upgrade head` 会报 "Can't locate revision '3b8a0d9f3f5a'",直接卡住 | 按 §4 步骤 3 复制文件 |
| 3 | **升级前必须做 pg_dump 备份** | 升级过程中任何一步出错都可能导致数据丢失,pg_dump 是唯一可靠的回滚手段 | 按 §4 步骤 1 备份 |

## §3 保留数据 vs 不保留数据:对比表

| 维度 | 保留数据(本文档) | 不保留数据(走主文档 §第二步) |
|------|------------------|------------------------------|
| 旧用户/面试/简历 | ✅ 完全保留 | ❌ 全部清空 |
| 操作步骤 | 8 步(含备份、迁移链修复) | 3 步(直接 `down -v` + 启动) |
| 是否需要备份 | **必须**(pg_dump) | 不需要 |
| 迁移链处理 | 手动复制旧迁移文件 | 自动重建 |
| 管理员账号 | 保留旧 admin,无需重建 | 必须重新跑 `create_first_admin.py` |
| 风险等级 | 中(需注意 3 条红线) | 低(从零开始) |
| 适用人群 | 已上线、有真实数据 | 测试环境、首次部署 |
| 完成时间 | ~30 分钟 | ~15 分钟 |

> **决策建议**:有真实用户数据 → 保留;纯测试 → 不保留。

## §4 完整升级步骤(主路径,8 步)

### 步骤 1:备份旧数据库(必须)

```bash
# 在 VM 上操作
# 1) 确认旧容器正在运行
sudo docker ps | grep ai-interview-postgres

# 2) 备份数据库到宿主机(命名带日期)
sudo docker exec ai-interview-postgres pg_dump -U demo -d ai_interview > ~/ai_interview_backup_$(date +%Y%m%d_%H%M%S).sql

# 3) 验证备份文件大小(空文件 < 1KB 是异常)
ls -lh ~/ai_interview_backup_*.sql
# 应该有几 MB 到几十 MB

# 4) 顺手记录一下旧数据库的关键数据行数(升级后对照用)
sudo docker exec -it ai-interview-postgres psql -U demo -d ai_interview << 'EOF'
SELECT 'users' AS table_name, COUNT(*) AS row_count FROM users
UNION ALL SELECT 'admins', COUNT(*) FROM admins
UNION ALL SELECT 'interviews', COUNT(*) FROM interviews
UNION ALL SELECT 'interview_messages', COUNT(*) FROM interview_messages
UNION ALL SELECT 'resumes', COUNT(*) FROM resumes;
EOF
```

> 💡 **为什么必须备份**:升级过程中迁移文件执行失败、镜像不兼容、卷挂载出错都可能导致数据库损坏。pg_dump 备份是回滚的唯一可靠手段。

### 步骤 2:停止旧容器(不删卷)

```bash
# 切换到新项目目录(假设共享文件夹已挂载)
cd /mnt/hgfs/ai-interview-backend

# 注意:绝对不能加 -v,否则数据库卷会被删,数据全丢
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml down

# 验证容器都停了,但卷还在
sudo docker ps -a | grep ai-interview
# 应该看到 STATUS 为 Exited

sudo docker volume ls | grep -E "postgres-data|redis-data"
# 应该看到 ai-interview-backend_postgres-data 和 ai-interview-backend_redis-data
```

> 💡 **`down` vs `down -v`**:`down` 只停容器、删容器、删网络,**保留卷**(数据安全);`down -v` 还会删卷,数据真丢。

### 步骤 3:复制旧迁移文件到新版(关键)

新版迁移链缺旧版的 `3b8a0d9f3f5a_add_resume_and_interview_tables.py`,必须把它复制过来,否则 alembic 找不到 revision 会报错。

```bash
# 旧版迁移文件路径(假设旧项目也在 /mnt/hgfs/ 下,或 Windows 共享目录里)
OLD_MIGRATION="/mnt/hgfs/ai-interview-backend-OLD/migrations/versions/3b8a0d9f3f5a_add_resume_and_interview_tables.py"

# 确认文件存在
ls -lh "$OLD_MIGRATION"

# 复制到新版
cp "$OLD_MIGRATION" /mnt/hgfs/ai-interview-backend/migrations/versions/

# 验证
ls /mnt/hgfs/ai-interview-backend/migrations/versions/
# 应该看到 7 个迁移文件:
#   e69999560f4f_init_table.py
#   ebf112457011_add_user_registration_fields.py
#   b4acc268ebc3_add_phone_country_code_field.py
#   c7f982abaaf0_add_waiting_list_table.py
#   3b8a0d9f3f5a_add_resume_and_interview_tables.py  ← 新复制的
#   d1e2f3a4b5c6_add_knowledge_base_tables.py
#   e2f3a4b5c6d7_add_position_templates.py
```

> ⚠️ 如果旧项目不在共享文件夹里,可以从 Windows 共享目录里找(`D:\AGI大模型\AI-project\ai-interview\ai-interview-backend\migrations\versions\`),用 `cp /mnt/hgfs/...` 或在 Windows 端直接复制。
>
> **备选方案**(给高级用户,跳过此步):见 §7,用 `alembic stamp` 直接改 `alembic_version` 表。

### 步骤 4:复制旧 .env 并补全新版必填项

```bash
# 1) 如果新版没有 .env,从模板创建
cd /mnt/hgfs/ai-interview-backend
cp -n .env.example .env   # -n:已有则不覆盖

# 2) 把旧版的关键配置迁移过来(逐项对比,确保不漏)
# 在 Windows 端用编辑器同时打开两个 .env:
#   旧:D:\AGI大模型\AI-project\ai-interview\ai-interview-backend\.env
#   新:D:\AGI大模型\ai-interview-agent\ai-interview-backend\.env
# 按 §6 表格逐项迁移
```

> 推荐在 Windows 端编辑器里操作(共享文件夹实时同步),比在 VM 终端里 nano 方便得多。

### 步骤 5:启动新版(自动用 pgvector 镜像接管旧卷)

```bash
cd /mnt/hgfs/ai-interview-backend

# 1) 先只启动 postgres,等它健康(让 pgvector 镜像接管旧数据目录)
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d postgres

# 2) 等待 postgres healthy(可能需要 10-30 秒,镜像升级数据目录)
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml ps postgres
# 看到 STATUS 为 Up (healthy) 再继续

# 3) 手动预装 vector 扩展(可选,迁移也会做,但提前更稳)
sudo docker exec -it ai-interview-postgres psql -U demo -d ai_interview -c "CREATE EXTENSION IF NOT EXISTS vector;"

# 4) 验证扩展已加载
sudo docker exec -it ai-interview-postgres psql -U demo -d ai_interview -c "SELECT extname FROM pg_extension WHERE extname='vector';"
# 应输出:
#   extname
#  ---------
#   vector
#  (1 row)

# 5) 启动所有服务
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d
```

> 💡 **为什么分两步启动**:pgvector 镜像第一次启动时会自动检测旧数据目录格式并升级(因为我们沿用了 `postgres-data` 卷)。只先起 postgres,等它健康再起 app,可以避免 app 因为 db 未就绪而反复重启。

### 步骤 6:跑迁移 + seed 岗位模板

```bash
# 1) 查看迁移版本(应该是 7 个,链尾 3b8a0d9f3f5a)
sudo docker exec -it ai-interview-app alembic current
# 应输出: 3b8a0d9f3f5a (head)

# 2) 升级到最新(自动执行 d1e2f3a4b5c6 + e2f3a4b5c6d7,建 RAG 表 + 岗位模板表)
sudo docker exec -it ai-interview-app alembic upgrade head

# 3) 验证表数(应 13 张:旧 9 张 + 新 4 张)
sudo docker exec -it ai-interview-postgres psql -U demo -d ai_interview -c "\dt"

# 4) 灌岗位模板(Agent 依赖,不灌则岗位匹配跑空)
sudo docker exec -it ai-interview-app python scripts/seed_position_templates.py

# 5) 验证岗位模板已写入
sudo docker exec -it ai-interview-postgres psql -U demo -d ai_interview -c "SELECT COUNT(*) FROM position_templates;"
# 应输出 10 左右
```

> ⚠️ **跳过 create_first_admin.py**:旧 admin 账号(`admin@ai-interview.com` / `ai-interview&admin`)仍在 `admins` 表里,无需重建。如果脚本是 `INSERT ... ON CONFLICT DO NOTHING`,跑了也无害;但**不要删 admins 表再重建**,否则旧管理员账号会丢。

### 步骤 7:健康检查

```bash
# API 健康检查
curl http://localhost:8006/api/v1/config/health
# 应输出: {"code":200,"message":"Success","data":{"status":"healthy"}}

# 看所有容器状态
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml ps
# 期望 5 个核心容器 Up:
#   ai-interview-app
#   ai-interview-celery-worker
#   ai-interview-celery-beat
#   ai-interview-postgres
#   ai-interview-redis
# (nginx 也会起,V1 不支持 profiles:donotstart,可忽略)
```

### 步骤 8:数据完整性对比

```bash
# 跑和步骤 1 完全一样的查询,对比行数是否一致
sudo docker exec -it ai-interview-postgres psql -U demo -d ai_interview << 'EOF'
SELECT 'users' AS table_name, COUNT(*) AS row_count FROM users
UNION ALL SELECT 'admins', COUNT(*) FROM admins
UNION ALL SELECT 'interviews', COUNT(*) FROM interviews
UNION ALL SELECT 'interview_messages', COUNT(*) FROM interview_messages
UNION ALL SELECT 'resumes', COUNT(*) FROM resumes
UNION ALL SELECT 'position_templates', COUNT(*) FROM position_templates;
EOF
```

行数应与步骤 1 完全一致(除了 `position_templates` 是新表,应该是 10 左右)。

## §7 备选方案:手动 stamp alembic(高级用户)

如果你不想复制旧迁移文件(比如想让新版 migrations 目录保持干净),可以手动改 `alembic_version` 表,跳过旧版独有的 `3b8a0d9f3f5a`:

```bash
# 1) 假设已按 §4 步骤 1-2 完成备份和停服

# 2) 不复制 3b8a0d9f3f5a,直接启动 postgres
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d postgres

# 3) 把 alembic_version 从 3b8a0d9f3f5a 改为 c7f982abaaf0(跳过一步)
sudo docker exec -it ai-interview-postgres psql -U demo -d ai_interview -c "UPDATE alembic_version SET version_num='c7f982abaaf0';"

# 4) 启动 app,跑迁移(会从 c7f982abaaf0 继续往上跑 d1e2f3a4b5c6 和 e2f3a4b5c6d7)
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d
sudo docker exec -it ai-interview-app alembic upgrade head

# 5) 验证
sudo docker exec -it ai-interview-app alembic current
# 应输出: e2f3a4b5c6d7 (head)
```

> ⚠️ **风险提示**:alembic 不会校验旧库表结构是否真的和新版迁移文件预期的一致。本项目两版这 7 张表字段**字节级一致**,所以安全;但如果以后再升级,建议改回"复制迁移文件"的主路径,更可控。

## §8 回滚方案

如果升级过程中出错,**先停止所有操作**,按下面步骤回滚:

### 场景 A:数据没丢,只想回到旧版

```bash
# 1) 停新版容器
cd /mnt/hgfs/ai-interview-backend
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml down

# 2) 启动旧版容器(假设旧项目目录还在 /mnt/hgfs/ai-interview-backend-OLD/)
cd /mnt/hgfs/ai-interview-backend-OLD
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d

# 3) 验证旧数据完好
curl http://localhost:8006/api/v1/config/health
```

### 场景 B:数据损坏,需要从备份恢复

```bash
# 1) 停所有容器
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml down -v
# ⚠️ 这次可以加 -v,因为我们要用备份重建

# 2) 启动 postgres(空数据)
cd /mnt/hgfs/ai-interview-backend
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d postgres
sleep 10  # 等 PG 就绪

# 3) 导入备份
sudo docker exec -i ai-interview-postgres psql -U demo -d ai_interview < ~/ai_interview_backup_20250624_120000.sql
# 注意 -i(不是 -it),从 stdin 导入

# 4) 启动所有服务
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d

# 5) 验证
sudo docker exec -it ai-interview-postgres psql -U demo -d ai_interview -c "SELECT COUNT(*) FROM users;"
# 行数应与备份前一致
```

> 💡 **回滚后的去向**:回滚后会回到旧版状态(9 张表 + 普通 postgres 镜像)。如果想再试一次升级,从头跑 §4 即可。

## §9 常见问题

### Q1:升级时报 `Can't locate revision '3b8a0d9f3f5a'`

**原因**:漏了 §4 步骤 3,旧迁移文件没复制。

**解决**:
```bash
# 补复制文件,然后让 alembic 重新跑
cp /mnt/hgfs/ai-interview-backend-OLD/migrations/versions/3b8a0d9f3f5a_add_resume_and_interview_tables.py /mnt/hgfs/ai-interview-backend/migrations/versions/
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml restart app
sudo docker exec -it ai-interview-app alembic upgrade head
```

### Q2:升级时报 `type 'vector' does not exist`

**原因**:postgres 镜像不是 pgvector。

**解决**:
```bash
# 1) 确认镜像
sudo docker inspect ai-interview-postgres | grep Image
# 应输出 pgvector/pgvector:pg16

# 2) 如果不是,说明用错 compose 文件了。停掉重来:
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml down
# 确认当前目录的 docker-compose.yml 里 image 是 pgvector/pgvector:pg16
grep "image: pg" docker-compose.yml docker-compose.dev.yml
# 应看到 pgvector/pgvector:pg16
```

### Q9:升级过程中 pgvector 镜像接管旧数据目录失败,postgres 容器一直 restarting?

**原因**:旧数据目录可能用了某些 pgvector 不识别的扩展(比如旧的 `pg_stat_statements` 状态),极少数情况会卡住。

**解决**:
```bash
# 1) 看具体报错
sudo docker logs ai-interview-postgres --tail 50

# 2) 如果是数据目录不兼容,最干净的方案是清场后用备份恢复(场景 B)
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml down -v
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d postgres
sleep 10
sudo docker exec -i ai-interview-postgres psql -U demo -d ai_interview < ~/ai_interview_backup_*.sql
sudo docker exec -i ai-interview-postgres psql -U demo -d ai_interview -c "CREATE EXTENSION IF NOT EXISTS vector;"
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d
sudo docker exec -it ai-interview-app alembic upgrade head
sudo docker exec -it ai-interview-app python scripts/seed_position_templates.py
```

### Q10:升级后 V1 docker-compose 报错 `profiles` 不识别,nginx 起不来?

**答**:不影响核心功能,nginx 是反代,不用也能跑。如需关掉,手动 `docker stop ai-interview-nginx` 即可。
