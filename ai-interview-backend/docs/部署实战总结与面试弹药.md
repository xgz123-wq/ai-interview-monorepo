## 二、项目概况

| 维度 | 内容 |
|------|------|
| 项目名 | ai-interview-agent（智面 · Agent 版） |
| 架构 | 三仓多包：FastAPI 后端 + Vue3 用户端 + Vue3 管理端 |
| 后端栈 | FastAPI + SQLAlchemy 2.x + PostgreSQL (pgvector) + Redis + Celery + LangChain + DeepSeek + DashScope |

### 问题 1：`parent snapshot does not exist`（build 失败，3 个 service 同时报错）

**现象**  
第一次跑 `docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d --build`：

```
✔ Container ai-interview-postgres    Created
✔ Container ai-interview-redis       Started
✔ Container ai-interview-app         Created
✔ Container ai-interview-celery-beat Created
✔ Container ai-interview-celery-worker Created
ERROR [celery-beat] exporting to image
ERROR [celery-worker] exporting to image
failed to prepare extraction snapshot ... parent snapshot ... does not exist: not found
```

3 个 Python service 跑完所有 RUN 步骤后，在 `exporting to image` 阶段同时失败，2 个 celery 报父 snapshot 不存在。

**根因分析**

3 个 service 共享同一个 image tag `ai-interview:dev`，并行 build 时：
- 第一个 build 完开始 export，把 intermediate image 写到临时名
- 第二个 build 的 export 时去引用第一个的 snapshot，但 Docker 正在清理第一个的临时 image
- 引用链断裂 → 报 "parent snapshot does not exist"

**深层根因**：`docker-compose.yml` 里 `app / celery-worker / celery-beat` 三个 service 都有 `build: .`，与 `docker-compose.dev.yml` merge 时，dev yml 里的 `image: ai-interview:dev` 不会**清除**主 yml 的 `build:`，结果 3 个 service 都要 build 同一个 tag 并行 export。

**3 个修法对比**

| 方案 | 操作 | 优点 | 缺点 |
|------|------|------|------|
| A | `docker compose build app` 单独 build 一个再 `up -d` | 不改任何文件，2 分钟搞定 | 每次部署都要先 build app 再 up，无法用 `up -d --build` 一条命令 |
| B | `docker builder prune -af` 清 BuildKit cache 后再 `up -d --build` | 干净状态重新 build | 慢（5-15 分钟），cache 失效 |
| **C** | 在 `docker-compose.dev.yml` 给 `celery-worker / celery-beat` 加 `build: !reset null` 显式重置 build 字段，让 celery 复用 app 镜像 | 根因直接命中，3 个 service 只 build 一次，体积减半，构建减半。**一劳永逸** | 需要 docker compose v2.20+ 支持 `!reset` 标签 |

**选择的方案 + 理由**：方案 C。从根上修了 compose merge 行为问题，未来部署都用 `up -d --build` 一条命令搞定。

**实施**：
```yaml
# docker-compose.dev.yml celery-worker 块加：
build: !reset null
# docker-compose.dev.yml celery-beat 块加：
build: !reset null
```

**验证**：3 个 Python 容器最终 IMAGE 都是 `ai-interview:dev`（同一 tag），证明 celery 没自己 build。

### 问题 2：`5434 address already in use`（iptables 残留）

**现象**  
方案 C 修复后第一次 up，postgres 容器创建时：
```
Error response from daemon: failed to set up container networking: driver failed
programming external connectivity on endpoint ai-interview-postgres: failed to
bind host port 0.0.0.0:5434/tcp: address already in use
```

app / celery 也连带失败（依赖 postgres healthy）。

**根因分析**

| 检查 | 结果 | 含义 |
|------|------|------|
| `ss -tlnp \| grep 5434` | 空 | 主机没进程 LISTEN 5434 |
| `systemctl status postgresql` | 空 | 没装 PG |
| `which psql` / `ls /etc/postgresql` | 空 | 同上 |
| `docker ps -a` | 1 个残留 ai-interview-postgres | 上次失败留下的 Created 容器 |

主机、容器、PG 服务**都正常**，但 docker 报端口占用。  
**真实根因**：docker daemon 内部 iptables 规则残留。`docker-compose down` 只删容器，不一定把 iptables nat 表的端口转发规则清干净。

**3 个修法对比**

| 方案 | 操作 | 风险 |
|------|------|------|
| **A** | `sudo systemctl restart docker` 彻底清 iptables | 低（停当前所有容器，但 volume 数据安全） |
| B | 改 `.env` 的 `POSTGRES_EXTERNAL_PORT=5434` → `5435` | 低（避开冲突） |
| C | `sudo iptables -t nat -F` 手工清规则 | 高（断所有容器网络） |

**选择的方案 + 理由**：方案 A（先试），10 秒搞定，root cause 直接命中。如果 A 不行再走 B 改端口。

**实施**：
```bash
sudo systemctl restart docker
cd /mnt/hgfs/ai-interview-backend
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml down
sudo docker-compose -f docker-compose.yml -f docker-compose.dev.yml up -d --build
```

**验证**：5 个容器全部 Up，postgres 5434 端口绑定成功。

### 问题 3：5434 端口长期占用（iptables 重启后又冲突）

**现象**  
重启 docker 后短期内 5434 不冲突，但**实际有持续占用的进程**（iptables 残留清掉了，但某个用户级服务或 VMware 内置服务还在占 5434）。

**根因**  
主机上 5434 实际有进程占用（可能 VMware 某个组件、可能主机 PG 服务），重启 docker daemon 只是清掉 docker 自己的 iptables 规则，不影响主机其他进程。

**修法选择**  
既然"避不开"5434 占用，**改 compose 端口映射**到 5435 是最稳的修法，不依赖调试端口占用源（调起来费时且收益小）。

**实施**：
```yaml
# docker-compose.yml postgres 块
ports:
  - "5435:5432"   # 改 5434 -> 5435
```

**同步更新部署文档**：`部署文档.md` § 端口说明 5434 → 5435，Navicat/DBeaver 连接提示同步。

**commit 留痕**：`fix: 宿主机 5434 端口占用，PG 主机端口改 5435`

### 问题 4：app 容器漏 `networks: - app-network`（DNS 解析失败）

**现象**  
app 容器健康检查端点 `/api/v1/config/health` 返回 503，body 显示：
```json
{"code":503,"message":"Service unhealthy","data":{"status":"unhealthy","services":{"api":"up","database":"down","redis":"up"}}}
```

**`redis: up` 但 `database: down`** —— 这就是关键线索，redis 通说明容器网络基本工作，PostgreSQL 不通。

**根因分析（5 步排查）**

1. **看现象**：app health 端点 503 + database: down
2. **定位置**：`app/api/client/v1/config.py:25-31` 跑 `SELECT 1` 抛异常 → 被 except 吞掉 → 503
3. **查上下文**：app 用 SQLAlchemy async + 环境变量 `POSTGRES_*` 拼连接串，docker-compose 用同名变量启动 postgres
4. **找根因**：手动跑 `docker exec app python` 用 asyncpg 连 PG，**真实错误浮出水面**：
   ```
   FAIL: gaierror: [Errno -2] Name or service not known
   ```
   DNS 解析不到 `postgres`！**app 容器在错误的网络里**。
5. **验证**：`docker inspect ai-interview-app` 看实际加入的网络 — 没有 `app-network`。

**深层根因（项目代码 bug）**

7 个 service 中，**只有 app 一个没声明 networks**：
| service | networks 字段 |
|---|---|
| app | ❌ 缺 |
| celery-worker / celery-beat / postgres / redis / nginx / flower | ✓ `networks: - app-network` |

dev yml 给 app 加了 `networks: - app-network`，但用户**先跑了主 yml 创建容器残留**，merge 后 up 没触发重建，dev yml 的 networks 没生效。

**修法**：在主 yml app 块 `depends_on` 后加：
```yaml
networks:
  - app-network
```

**commit 留痕**：`fix: app 服务补 networks: - app-network（漏配导致 DNS 解析不到 postgres）`

### 问题 5：image 级 HEALTHCHECK 让 celery 误判 unhealthy

**现象**  
方案 C 修复 + networks 修复后，app 起来了 (healthy)，但 **celery-worker / celery-beat 也显示 (unhealthy)**，虽然它们的进程在跑、消息正常处理。

**根因分析**

`Dockerfile.dev` 第 54-55 行：
```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8006/api/v1/config/health || exit 1
```

**这条写在 image 里，对所有用该 image 启动的容器都生效**：
- `app`（`python main.py`）→ 监听 8006 → curl 成功 → healthy
- `celery-worker`（`celery worker`）→ **不监听 8006** → curl connection refused → unhealthy
- `celery-beat`（`celery beat`）→ **不监听 8006** → connection refused → unhealthy

`docker compose` 的 `command:` 只能覆盖 `CMD`，**覆盖不了 HEALTHCHECK** — 必须用 service-level `healthcheck:` 显式覆盖。

**为什么方案 C 让这个 bug 显形了**  
之前 celery-* 各 build 各的 image，副作用被"误打误撞"稀释（多份 image 共享 layer 但 head 隔离）。方案 C 让 3 个 service **共享同一 image tag** 后，bug 集中暴露。

**修法选择（2 个）**

| 方案 | 操作 | 影响 |
|------|------|------|
| **A** | 删 `Dockerfile.dev` 的 HEALTHCHECK 指令 | app 由 compose 级 healthcheck 兜底（已存在），celery 不再被拖累 |
| B | 在 docker-compose.yml 给 celery-* 各加 service-level healthcheck | 保留 image 通用性，但 `/proc/*/cmdline` 扫描非标准模式 |

**选择的方案 + 理由**：方案 A。从根上修了 image HEALTHCHECK 设计缺陷，改动小（删 2 行 + 注释说明）。保留 image 通用性（不被单一用途的 healthcheck 绑死）。

**实施**：
```dockerfile
# Dockerfile.dev 删除：
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8006/api/v1/config/health || exit 1
```

**commit 留痕**：`chore: 删 image 级 HEALTHCHECK`

### 问题 6：postgres 容器双绑 5434 + 5435

**现象**  
改端口到 5435 后重启，postgres 容器：
```
0.0.0.0:5434->5432/tcp, 0.0.0.0:5435->5432/tcp, [::]:5434->5432/tcp, [::]:5435->5432/tcp
```

一个容器绑了两个主机端口（IPv4 + IPv6 各 2 个 = 4 个绑定）。

**根因分析**

`docker-compose.dev.yml` line 15：
```yaml
ports:
  - "${POSTGRES_EXTERNAL_PORT:-5434}:5432"
```

docker compose merge 时，**ports 是 list 类型，两个 yml 的 ports 会追加**。我改了主 yml 到 5435，但 dev yml 的 `${POSTGRES_EXTERNAL_PORT:-5434}:5432` **默认展开为 5434**，最终 ports list 是 `["5435:5432", "5434:5432"]` → 双绑。

**修法**：删 dev yml 的 ports 字段，让 merge 后只剩主 yml 的 5435。

**commit 留痕**：`fix: dev yml 删除 postgres ports 字段（避免 merge 后双绑 5434+5435）`

### 问题 7：celery-beat healthcheck 报 `ps: not found`

**现象**  
给 celery-beat 加 healthcheck 后，状态变 `(unhealthy)`：
```yaml
test: ["CMD-SHELL", "ps aux | grep -E 'celery.*beat' | grep -v grep || exit 1"]
```

但 celery-beat 日志完全正常，每分钟发一次任务：
```
[2026-06-24 14:19:44] Scheduler: Sending due task demo-every-minute
```

**根因分析**

手动跑 healthcheck 命令：
```bash
sudo docker exec ai-interview-celery-beat sh -c "ps aux | grep -E 'celery.*beat' | grep -v grep || echo NO_MATCH"
```
输出：
```
sh: 1: ps: not found
NO_MATCH
```

**`python:3.12-slim` 镜像默认不安装 `procps`**（ps 工具的包）。celery beat 进程完全正常，只是 healthcheck 命令用了容器里没有的 `ps`。

**修法**：不依赖 ps/pgrep，扫 `/proc/*/cmdline` 找进程（Linux 内核导出，不需要任何工具）：

```yaml
healthcheck:
  test: ["CMD-SHELL", "grep -la 'celery.*beat' /proc/[0-9]*/cmdline 2>/dev/null | grep -q . || exit 1"]
  interval: 30s
  timeout: 10s
  retries: 3
  start_period: 60s
```

- `grep -la` — 列出匹配的文件名（`-a` 强制把二进制 cmdline 当文本扫）
- `/proc/[0-9]*/cmdline` — 所有进程的 cmdline
- `grep -q .` — 检查输出非空
- 空 → `exit 1`；非空 → 0（healthy）

**同时给 celery-worker 加 healthcheck**：
```yaml
healthcheck:
  test: ["CMD-SHELL", "celery -A app.core.celery_app inspect ping -t 5 || exit 1"]
  interval: 30s
  timeout: 10s
  retries: 3
  start_period: 60s
```
- worker 用 `celery inspect ping`（worker 主动响应 ping）
- beat 用 `ps /proc 扫描`（beat 是调度器，不响应 ping）

**commit 留痕**：`chore: 给 celery-* 加专属 healthcheck（worker 用 inspect ping, beat 用 /proc 扫描）`

### 问题 8：3 张核心业务表缺失迁移（项目代码 bug）

**现象**  
执行 `alembic upgrade head` 后数据库只有 10 张表，但部署文档说应该有 13 张（含 `interviews / interview_messages / resumes`）。  
后来实际跑完 13 张都有了，但 alembic_version 表指向 `4c323f793036`（文件系统里找不到的 revision）。

**根因分析**

读 ORM 模型（`app/models/`）：
- `app/models/interview.py:7` — `Interview` 模型，`__tablename__ = "interviews"`
- `app/models/interview_message.py:6` — `InterviewMessage` 模型，`__tablename__ = "interview_messages"`
- `app/models/resume.py:6` — `Resume` 模型，`__tablename__ = "resumes"`

读迁移历史（`migrations/versions/`）：
- `e69999560f4f_init_table.py` 实际只建 4 张表（admins / users / admin_tokens / tokens）
- 后续 5 个迁移加了 waiting_list / knowledge_* / question_bank / position_templates
- **`interviews / interview_messages / resumes` 从来没在任何迁移里建过**

**但部署文档 § 2.3.2 却声称 init_table 建了 7 张表（包括这 3 张）** — 文档与代码不一致，3 张表是文档里的"幽灵表"。

**外键依赖**：
- `interviews.resume_id` → `resumes.id`（先建 resumes）
- `interview_messages.interview_id` → `interviews.id`（再建 interviews，最后建 interview_messages）

**修法**：手写 3 个 alembic 迁移文件，按依赖顺序串成新链：
```
e2f3a4b5c6d7 (add_position_templates)
  └─ b1a2c3d4e5f6 (add_resumes_table)
       └─ b2a3c4d5e6f7 (add_interviews_table)
            └─ b3a4c5d6e7f8 (add_interview_messages_table)  ← 新 head
```

**每个迁移独立 commit**（符合 alembic 习惯，便于回滚）。

**alembic_version 对齐**：直接 `UPDATE alembic_version SET version_num='b3a4c5d6e7f8'`，因为旧值 `4c323f793036` 在文件系统里不存在，`alembic stamp` 也会失败。

**commit 留痕**：
- `chore: 加 resumes 迁移`
- `chore: 加 interviews 迁移`
- `chore: 加 interview_messages 迁移`
- `fix: 部署文档迁移章节与实际代码对齐`

## 四、面试弹药（STAR 法则 + 4 个最亮点）

> 面试官高频问："你部署这个项目时踩过什么坑？怎么解决的？"  
> 用 STAR 法则（Situation / Task / Action / Result）回答，挑 3-4 个最有亮点的讲。

### STAR 1：docker compose merge 行为导致 celery 重复 build

> **S（Situation）**：第一次跑 `docker compose up -d --build`，3 个 Python service 都在 build 同一 image tag `ai-interview:dev`，并行 export 时 `parent snapshot does not exist`，build 失败。  
> **T（Task）**：定位 root cause，找到根因。  
> **A（Action）**：对比 3 个修法（单独 build / 清 cache / 改 yml），最终选方案 C——在 `docker-compose.dev.yml` 给 celery 加 `build: !reset null`（v2.20+ 特性），显式清空从主 yml merge 来的 `build:` 字段，让 celery 复用 app 镜像。  
> **R（Result）**：3 个 Python 容器共享同一 image tag，build 一次复用，体积减半、构建时间减半。  
> **为什么这么设计**：根因是 `docker-compose.yml` 和 `docker-compose.dev.yml` merge 时，dev yml 的 `image:` 不会**清除**主 yml 的 `build:`（merge 默认行为是补充不是覆盖）。`!reset null` 是 v2.20+ 引入的 YAML tag，专门用来"重置 merge 来的字段为 null"。

### STAR 2：app 容器漏 networks 导致 DNS 解析不到 postgres

> **S**：app 一直 (unhealthy)，health 端点返回 503，body 显示 `database: down` 但 `redis: up`。  
> **T**：5 步排查（看现象 → 定位置 → 查上下文 → 找根因 → 验证）。  
> **A**：手动跑 `docker exec app python` 用 asyncpg 连 PG，**真实错误浮出水面**：`gaierror: Name or service not known`。检查 `docker inspect ai-interview-app` 发现 app 没加入 `app-network`。看 7 个 service 的 yml 配置，**只有 app 一个漏了 `networks: - app-network`** —— 这是项目代码的真实 bug。补 networks 字段 + `docker compose down && up -d --build` 强重。  
> **R**：app 起来 (healthy)，整个部署链路打通。  
> **为什么这么设计**：Docker 容器通过 service name 解析其他 service 依赖 Docker 内置 DNS（仅在同一 network 内有效）。app 不在 `app-network` 就解析不到 `postgres` 服务名，asyncpg 直接 connection refused。  
> **加分点**：5 步排查法 + 手动构造最简复现 + 对比 7 个 service 配置找异常 —— 这是 systematic debugging 思维。

### STAR 3：image HEALTHCHECK 设计失误 + slim 镜像无 ps

> **S**：方案 C 让 celery 共享 app image 后，celery 显示 (unhealthy)，但日志完全正常（每分钟发一次任务）。  
> **T**：找根因。  
> **A**：检查 `Dockerfile.dev` 第 54 行有 `HEALTHCHECK curl http://localhost:8006/...`。celery 不监听 8006 必然 connection refused。`docker compose` 的 `command:` 覆盖不了 `HEALTHCHECK`，必须 service-level 显式覆盖。删 image 级 HEALTHCHECK + 给 celery-worker 加 `celery inspect ping`、celery-beat 用 `/proc/*/cmdline` 扫描（因为 `python:3.12-slim` 镜像没装 `procps` 包，`ps` 不可用）。  
> **R**：5 个容器全部 (healthy)，celery 也正确显示状态。  
> **为什么这么设计**：  
> - 同一个 image 应支持多角色（app / celery-worker / celery-beat），绑死单一用途的 healthcheck 是设计缺陷。  
> - `/proc/<pid>/cmdline` 是 Linux 内核导出的进程命令行，**不依赖任何工具**，可移植性比 ps/pgrep 强。  
> - slim 镜像 = 生产首选（小体积），但很多 GNU 工具被裁掉，需要知道"哪些能用什么不能"。

### STAR 4：alembic 迁移链断裂 + 文档与代码不一致

> **S**：`alembic current` 报 `Can't locate revision identified by '4c323f793036'`，但 13 张表都建好了。

## 五、Know-How 总结（深度理解）

### 1. Docker Compose merge 行为（最容易踩的坑）

| 字段类型 | merge 行为 |
|---|---|
| **标量**（string/int） | 后 yml 覆盖前 yml |
| **list / array** | 两边 list 追加 |
| **map** | 两边 map 合并（同 key 后 yml 覆盖） |

**这意味着什么**：
- `image:` 是标量 → dev yml 的 `image: ai-interview:dev` 覆盖主 yml 的 `image` 字段（但 dev yml 没写 image 时用主 yml 的）
- `build:` 是 map → 两边合并（可能重复或冲突）
- `ports:` 是 list → 两边 **追加**（这是问题 6 的根因）
- `environment:` 是 list → 两边追加
- `volumes:` 是 list → 两边追加
- `networks:` 是 list → 两边追加（**但 service 实际只加入有显式 networks 字段时指定的 network**）

**解决方案**：
- `!reset null` 标签：v2.20+ 引入，**显式重置** merge 来的字段为 null
- 别名 alias 机制：用 yml anchors + overrides
- 改设计：尽量在主 yml 写全，dev yml 只做环境特定覆盖

### 2. BuildKit Cache 行为

- `docker buildx` / BuildKit 23+ 用 layered snapshot 机制
- 每个 layer 是一个 snapshot，build 时引用 parent snapshot
- **删 image 不清 cache**：`docker image rm` 删的是 image 标签，BuildKit cache（`/var/lib/docker/buildkit/`）里可能还有 snapshot 引用
- 并发构建撞 snapshot：3 个 build 共享同一 image tag 时，第一个 build 完清理 intermediate image，第二个的 export 引用链断了
- **修法**：`docker builder prune -af` 清 BuildKit cache（不删 image / volume）

### 3. Docker 网络 & DNS

- 同一 network 内的容器可以通过 service name 互相解析（Docker 内置 DNS）
- 容器加入多 network 时，每个 network 内都能解析该 network 内的 service
- **service 必须显式声明 `networks: - xxx`，否则只加入默认网络**（compose 自动创建的 `<project>_default`）
- iptables 残留：`docker-compose down` 不一定清 iptables 规则，需要 `systemctl restart docker` 彻底清
- 端口映射是**主机端口 → 容器端口**，主机端口冲突需要改 compose；容器端口冲突要看 PG 自己的 `pg_hba.conf` 监听设置

### 4. Alembic 迁移链

- `alembic_version.version_num` 是**单字段**，记录当前 migration 的 revision
- 必须和 `migrations/versions/*.py` 里某个文件顶部的 `revision: str = 'xxx'` **严格匹配**
- 文件被删/重命名 → alembic 找不到 → `alembic current` 报 `Can't locate revision`
- `alembic stamp <revision>` 是规范修法（校验 + 更新），但找不到的 revision 也会失败
- **极端修法**：直接 `UPDATE alembic_version SET version_num='<new_head>'`（只在规范方法都不行时用）
- 迁移文件应**按依赖顺序串成链**（down_revision 指向上一个），`alembic history` 会从 base 到 head 完整列出
- **外键依赖决定建表顺序**：A 表有 FK → B 表，则必须先建 B 表
- 项目里常用 `--autogenerate` 自动生成迁移，但 auto 不能捕获所有变更（rename / 类型变更 / server_default 变化），需要人工 review

### 5. Image 级 vs Service 级 HEALTHCHECK

| 类型 | 配置位置 | 作用范围 | 能否被 service 级覆盖 |
|---|---|---|---|
| Image 级 | `Dockerfile.HEALTHCHECK` | 所有用该 image 的容器 | ✅ service 级 healthcheck 会覆盖 |
| Service 级 | `docker-compose.yml healthcheck:` | 仅该 service | — |

**常见错误**：
- 写 image 级 HEALTHCHECK 绑死 8006 → 所有用该 image 的 service 都被影响
- 同一 image 多种角色（app + celery）时，必须删 image 级 + 给每个 service 写专属
- `command:` 只能覆盖 `CMD`，覆盖不了 `HEALTHCHECK`

### 6. /proc 文件系统（debug 神器）

- `/proc/<pid>/cmdline` — 进程命令行（用 `\0` 分隔参数，**不依赖 ps / pgrep**）
- `/proc/<pid>/status` — 进程状态
- `/proc/<pid>/fd/` — 进程打开的文件描述符
- **slim 镜像**（如 `python:3.12-slim`）默认没装 `procps`（ps / pgrep / top），但 `/proc` 一直可用
- healthcheck 写 `/proc/*/cmdline` 扫描比 `ps aux` 更可移植

### 7. 端口冲突排查 SOP

| 步骤 | 命令 | 期望 |
|---|---|---|
| 1 | `ss -tlnp \| grep <port>` | 看到 LISTEN 进程 → 杀 / 改端口 |
| 2 | `systemctl status postgresql` | 看到 active → 停服务 |
| 3 | `docker ps -a \| grep <port>` | 看到残留容器 → `docker rm -f` |
| 4 | `sudo systemctl restart docker` | 清 iptables 残留 |
| 5 | 改 compose 端口映射 | 终极方案，绕开冲突 |

## 九、给同事的操作建议

1. **首次部署**严格按本文档第七章"部署自检清单"跑，不要跳步
